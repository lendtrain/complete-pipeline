## PLAN CREATION RULES

1. **Every section must have a recommended agent.** Match sections to the best specialist from the inventory above. If a section is simple enough for the main agent, say "main agent."

2. **Every section must have verification commands.** Not "check that it works" тАФ actual terminal commands with expected output.

3. **The first section MUST produce a running/verifiable artifact.** A scaffold that starts, a plugin that installs, a CLI that responds тАФ something observable.

4. **Include the parallel execution map.** This is critical for Boris Laws compliance. Which sections can run in separate worktrees simultaneously?

5. **Include the agent delegation summary.** This gives the orchestrator a quick reference for spawning agents without re-reading the full plan.

6. **Match skills to sections.** If `frontend-design`, `mcp-builder`, `test-driven-development`, or any other skill would help a section, note it.

7. **Scope each section to ~30 minutes of focused work.** If a section is larger, break it into sub-sections.

8. **Include "How to Debug" per section.** What are the most likely failure points and how to resolve them?

9. **Follow the sequencing constraints from the planning prompt exactly.** Do not reorder unless you have a clear reason, documented in the plan.

10. **All config in .env or equivalent.** If the planning prompt specifies externalized config, ensure every section that touches config references .env, never hardcoded values.
